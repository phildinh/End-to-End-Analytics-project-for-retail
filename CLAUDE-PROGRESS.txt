  ARCHITECTURAL CHANGES (see ADR-009/010/011 in docs/architecture.md):
  - ADR-009: dbt now fully owns the mart schema. Dropped 6 leftover FK
    constraints from 03_constraints_indexes.sql that targeted dbt-managed
    mart tables (fk_subcategory_category, fk_product_subcategory,
    fk_sales_territory, fk_sales_calendar, fk_returns_territory,
    fk_returns_calendar) — `dbt seed` can't TRUNCATE a table referenced by
    an FK, and `dbt run`'s table rebuild (create-temp -> rename-old-to-backup
    -> create-new -> drop-backup) fails if another table FKs the backup.
    Also switched dim_territory/dim_customer/dim_product surrogate keys from
    `INT GENERATED ALWAYS AS IDENTITY` to `row_number()` in the model SQL,
    since CTAS doesn't preserve IDENTITY. database/03_constraints_indexes.sql
    is now a historical record only, not re-runnable as-is. CLAUDE.md and
    docs/architecture.md Section 5 updated to match.
  - ADR-010: fact_sales surrogate key changed to
    md5(OrderNumber || OrderLineItem || OrderDate) — found 242
    (OrderNumber, OrderLineItem) pairs reused across different OrderDates
    in the 2020-2026 extended dataset (0 collisions once OrderDate added
    across all 341,168 rows). CLAUDE.md grain table and
    docs/data_dictionary.md updated to match.
  - ADR-011: fact_returns now aggregates source rows by
    (ReturnDate, TerritoryKey, ProductKey), summing ReturnQuantity — found
    1 duplicate combo (2024-01-24, territory 3, product 185) with two rows
    of ReturnQuantity=1 each. docs/data_dictionary.md updated to note this.

  ARCHITECTURAL DECISION (see ADR-012 in docs/architecture.md):
  - The "adventureworks" warehouse DB stays on the local Windows Postgres 16
    instance (already loaded/tested in Phases 1-3) rather than being
    containerized. Airflow containers reach it via host.docker.internal,
    with DB_HOST overridden in docker-compose's environment (other DB_* vars
    come from ../.env via env_file). The docker-compose `postgres` service is
    Airflow's own metadata DB only.

  FIXES DURING docker-compose up / dbt debug:
  - pip resolved dbt-postgres==1.8.2's unconstrained dbt-core dependency to a
    pre-release 2.0.0a1, which can't parse profiles.yml correctly
    ("password: invalid type: integer"). Fixed by pinning dbt-core==1.8.8.
  - `dbt debug` failed the git dependency check (git not installed in the
    apache/airflow base image). Fixed by adding a `USER root` apt-get install
    git step to the Dockerfile before switching back to `USER airflow`.
  - load_to_staging failed with PermissionError on os.makedirs("/opt/airflow/
    project/logs") — that path wasn't a real mount, just an implicit
    (root-owned) directory created by Docker for the sibling mounts. Fixed by
    adding an explicit `../logs:/opt/airflow/project/logs` volume mount.

  TEST RESULTS:
  - docker-compose up: postgres, airflow-init, airflow-webserver,
    airflow-scheduler all healthy; `airflow dags list` shows
    adventureworks_pipeline with no import errors.
  - Unpausing the DAG fired two runs (a manual trigger plus an auto-scheduled
    run for the prior day, since catchup=False only skips runs before
    start_date). Both completed with all 5 tasks `success`.
  - dbt_run reported "INSERT 0 0" on both runs: load_incremental.py's
    default (no --file) re-appends the full historical CSVs, and the
    container's UTC clock produced a loaded_at older (numerically) than the
    naive +10-offset loaded_at already in mart from the Phase 3 host run —
    so the `loaded_at > max(loaded_at)` incremental filter matched nothing.
    mart row counts stayed at fact_sales=341168 / fact_returns=4496 across
    both runs (28/28 dbt tests pass both times) — confirms re-running the
    DAG produces no duplicates.
  - Cleanup: staging.fact_sales/fact_returns had accumulated multiple
    duplicate full-CSV batches from repeated test runs (this session and
    earlier). Re-ran load_full.py (truncate+reload) then `dbt run` from the
    host to reset staging to one clean batch; dbt run replaced all 341168 /
    4496 mart rows via delete+insert (INSERT 0 341168 / INSERT 0 4496) and
    28/28 tests still pass — confirms the reset was idempotent at the mart
    layer.

  KNOWN LIMITATION (not fixed — out of scope for Phase 4):
  - load_incremental.py's default behaviour (no --file) re-appends the
    entire historical CSV every run, so staging.fact_sales/fact_returns grow
    by a full duplicate batch on every nightly run. mart stays correct
    (delete+insert by deterministic md5 key), but staging will accumulate
    unbounded duplicates over time. Revisit in a future phase: either point
    the nightly DAG at a true daily-delta file, or have load_incremental.py
    deduplicate/prune superseded batches in staging.
